# プロダクト定義

Trinity のプロダクト像と判断の軸を言語化する文書である。開発標準プロセス（ [`docs/process.md`](process.md) ）におけるトリアージ・設計分岐・課題起票など、あらゆる採否判断はこの文書を単一の正として照合する。自律運転がどれだけ長く回ってもプロダクトの軸がブレないことを目的とし、本文書の変更は人間だけが行う。

## Vision

Trinity は、開発者の仕事を「プロンプトを打つこと」から「ループを設計すること」へ転換する。Claude Code のサブエージェントだけを使い、Issue の起票からリリースまでの開発プロセスを Production-Ready の品質水準で自律駆動するハーネスである。

単一エージェントに長時間タスクを任せると、コンテキストの膨張とともに計画がドリフトし、評価者が自分の成果物を甘く見るようになる。一方で品質を人間のレビューで担保しようとすると、人間が律速となり自動化の意味が失われる。Trinity は役割分離と独立評価によって品質を構造的に担保し、人間を軸の更新と例外処理に専念させることで、この2つの失敗モードを同時に解消する。

## Users

想定ユーザーは、Claude Code を日常的に使い、自分のリポジトリの定常的な開発をエージェントに任せたい個人開発者・小規模チームである。受け入れ基準を伴う Issue 文化と Conventional Commits が定着したリポジトリで最大の効果を発揮する。Trinity 自身のリポジトリがその最初の適用先であり、自律運転のドッグフーディング環境を兼ねる。

## Principles

プロダクトのあらゆる採否判断の基準となる原則を以下に示す。新機能・変更・自動起票される課題は、少なくとも1つの原則を強化し、かついずれの原則も毀損しないことを採択の必要条件とする。

| 原則 | 内容 |
| :-- | :-- |
| 評価の独立性 | 生成と評価を同一コンテキストに置かない。Evaluator は読み取り専用とし、受け渡しはファイル経由に限る。 |
| 品質非妥協 | Production-Ready 未満でループを抜けない。自動化の都合で品質基準を下げない。 |
| プロンプトが実体 | ビルド・依存・ツールチェーンを持たず、markdown の規約だけで挙動を定義する。 |
| チェックアウト不可侵 | 作業は隔離 worktree の中だけで行い、ユーザーのチェックアウトとリモートの履歴を壊さない。 |
| リーン | 規約は実在する失敗モードに対応する最小限だけを書く。over-spec と over-refactor を排する。 |
| 人間は設計と例外 | 定常運転はループが担う。人間は軸の更新・blocked の解消・リリースの監督に専念する。 |

## Scope

スコープの境界を以下に示す。「やらないこと」は機能不足ではなく設計判断であり、解除には本文書の改訂を要する。

### In Scope

Trinity が責務として引き受ける範囲は、開発プロセスそのものの自律駆動である。

- Issue の起票からリリースまでの開発標準プロセスの定義と無人駆動
- Planner / Generator / Evaluator のループによる実装と品質判定
- worktree・Pull Request・リリース運用の自動化（git-flow スキルと release-please への委譲を含む）

### Out of Scope

以下は外部の仕組みに委譲し、Trinity では作らない。

- CI/CD 基盤そのものの代替（GitHub Actions・release-please に委譲する）
- エージェント実行基盤の自作（Claude Code のサブエージェント機構に委譲する）
- git 運用・コードレビューの内製（git-flow スキル・code-review コマンドに委譲する）
- 複数リポジトリ横断のオーケストレーション、GUI・ダッシュボードの提供

## Roadmap

バックログ（自動起票を含む）の源泉となる中期テーマを優先順に示す。個別の Issue はいずれかのテーマに紐づくことを原則とし、どのテーマにも紐づかない提案はトリアージで人間の判断へ送る。

| テーマ | ねらい |
| :-- | :-- |
| 自律運転の信頼性 | 中断からの再開・チェックポイント・確認の集約・ガードレールなど、無人で回し続けるための堅牢化 |
| 評価の独立性と検証手段の強化 | ツール権限の最小化・視覚検証の導入など、判定の質と構造的独立性の向上 |
| 利用側オンボーディングの簡素化 | 前提の自動セットアップ・ドキュメント整備など、導入コストの最小化 |

## Governance

軸そのものの運用規約を以下に示す。

- 本文書の変更は人間だけが行う。自律運転は変更を提案する Issue の起票までとし、本文書を直接編集しない。
- 変更は Pull Request として記録し、影響に応じた Conventional Commits 接頭辞を付ける。
- 開発プロセス上の判断が本文書で判断できない場合、その判断は `status: blocked` として人間へ委ねる。
